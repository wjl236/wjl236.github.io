逻辑漏洞

  1. 绕过认证（Authentication Bypass）
    概念：攻击者通过绕过身份验证过程，直接访问应用程序中受保护的部分。
    常见漏洞：
    弱密码或简单认证：如明文存储的密码、使用默认密码等。
    URL篡改：通过修改请求中的参数绕过认证（例如，改变user_id或role参数）。
    Session管理不当：比如会话ID固定，或者用户认证信息未正确校验。
    防护：
    强制使用复杂密码策略。
    加密存储密码，使用bcrypt、argon2等安全算法。
    使用双因素认证（2FA）。
    定期检查Session和Token的有效性。
    防止会话固定，通过session_regenerate_id()生成新会话ID。
  2. 权限绕过（Authorization Bypass）
    概念：攻击者利用权限控制的漏洞，获取自己不应访问的资源。
    常见漏洞：
    URL/请求篡改：攻击者修改请求参数（如user_id=123）以访问其他用户的资源。
    文件权限控制不当：文件上传、访问权限设置错误，攻击者可以上传恶意文件或访问他人数据。
    防护：
    确保所有敏感操作都经过服务器端的权限检查。
    使用严格的角色访问控制（RBAC）机制来管理权限。
    检查和验证所有用户输入，防止篡改请求中的敏感参数。
    防止暴露系统文件和目录，通过/etc/passwd等文件检查来确保安全。
  3. 逻辑错误（Business Logic Flaw）
    概念：应用程序在实现某些业务逻辑时的设计不当，允许攻击者通过非预期方式操控应用程序。
    常见漏洞：
    付款系统漏洞：如通过篡改订单金额、频繁请求订单等方式欺骗支付系统。
    库存数量操控：通过篡改库存信息来获取不应获得的商品。
    竞拍漏洞：通过脚本自动化操作操控竞拍或价格。
    防护：
    彻底审查业务逻辑，确保每个流程的处理都符合预期。
    使用状态机和事务控制来确保操作的有效性与一致性。
    设置合理的限制和验证（如订单金额、支付次数、商品库存等）。
  4. 时间竞态（Race Condition）
    概念：攻击者通过并发操作，利用程序在执行过程中产生的竞态条件，从而绕过安全控制或得到非法的资源。
    常见漏洞：
    并发操作不一致：例如两个用户在几乎相同的时间内请求修改同一资源（如余额、库存等），导致程序没有正确处理这种并发情况。
    防护：
    在修改敏感数据时使用锁机制，确保并发操作时的一致性。
    对操作进行事务处理，确保数据一致性。
    避免对数据进行非原子操作。
  5. 不当的数据验证（Improper Data Validation）
    概念：攻击者通过提交未经验证的数据，达到绕过预期逻辑的目的。
    常见漏洞：
    客户端验证绕过：通过禁用JavaScript或直接发送恶意请求，绕过前端验证。
    不完全验证：如表单输入未进行彻底的合法性检查。
    文件上传漏洞：如上传不受限制的文件类型（例如PHP文件）。
    防护：
    严格的服务器端验证，不能依赖客户端验证。
    对所有用户输入进行彻底的合法性检查。
    使用白名单来限制上传的文件类型。
  6. 计数器/时间窃取（Time-based or Counter-based Attacks）
    概念：通过对应用程序响应时间或计数器等行为的观察，窃取数据或绕过逻辑。
    常见漏洞：
    验证码绕过：攻击者通过暴力破解验证码或基于时间的逻辑来绕过身份验证。
    Session时间滥用：通过操控Session过期时间或利用延迟操作。
    防护：
    在验证码或token生成中使用随机、不可预测的算法。
    使用更复杂的时间间隔来限制暴力破解。
    避免使用简单的基于时间的系统（如短时间间隔的会话失效机制）。
  7. 缓存投毒（Cache Poisoning）
    概念：攻击者通过操控应用程序的缓存，插入恶意内容，影响后续请求。
    常见漏洞：
    不当的缓存配置：缓存了敏感信息或未正确隔离不同用户的数据。
    防护：
    确保缓存中不包含敏感信息。

使用基于用户身份的缓存隔离策略。
8. URL重定向漏洞（Open Redirect）
概念：应用程序没有验证跳转目标，允许攻击者通过恶意URL欺骗用户跳转到不安全的网站。
常见漏洞：
URL中的参数没有过滤或校验，允许攻击者操控跳转地址。
防护：
验证跳转的目标URL，确保其为合法的预定义地址。
避免将用户输入的URL直接用于跳转。
9. 信息泄露（Information Disclosure）
概念：攻击者通过应用程序泄露信息（如错误信息、调试信息）来推测系统结构或漏洞。
常见漏洞：
错误信息泄露：详细的错误信息会泄露系统配置、数据库结构等。
防护：
禁止在生产环境中显示详细的错误信息。
通过日志记录和警告来追踪错误，但不将敏感信息暴露给用户。
10. 逻辑时序攻击（Time-of-check to Time-of-use (TOCTOU) Bugs）
概念：攻击者利用系统状态的变化，例如在检查某个条件和实际操作之间发生的状态变化来攻击系统。
常见漏洞：
文件操作：例如，检查文件是否存在时，文件状态可能在操作时发生变化，导致绕过安全检查。
防护：
确保文件操作是原子性的，避免中间状态变化。
使用锁机制和事务来确保操作的一致性。

web弱类型

  1. JavaScript的弱类型特性
    JavaScript 是一种弱类型语言，意味着它会自动将不同类型的数据进行转换，这可能会导致一些安全漏洞
    常见的弱类型特性：
    == 和 === 的差异：
    == 会进行类型转换（强制类型转换），而 === 会严格比较，不进行类型转换
    例如，’1’ == 1 会返回 true，但 ‘1’ === 1 会返回 false
    字符串与数字的自动转换：
    “” + 1 结果是 “1”
    1 + 1 结果是 2
    ‘1’ - 1 结果是 0，因为 JavaScript 会尝试将字符串转换成数字
    null 和 undefined 的比较：
    null == undefined 会返回 true，但是 null === undefined 会返回 false
    这种差异经常被利用在绕过条件判断时
    数组与对象的转换：
    空数组 [] 转换成 false，但 [] == 0 返回 true
    {} == [] 会返回 false，但 {} == {} 和 [] == [] 都会返回 false
    应用：
    弱类型特性常常被用来绕过输入验证、控制流、或执行不安全的操作。例如，利用 == 比较时，可能会绕过某些验证逻辑（如 if (input == 0)）

  2. PHP的隐式类型转换
    PHP 是一个弱类型的语言，执行运算时会自动转换变量类型。PHP的类型转换常常成为漏洞的来源
    常见的隐式类型转换：

字符串和数字转换：
‘0’ == 0 返回 true
‘1’ == true 返回 true，但 ‘1’ === true 返回 false
空字符串 “” 被视为 false，而非空字符串被视为 true
数组和对象与布尔值的转换：
空数组 [] 被转换为 false
array(1, 2) 被转换为 true
数字字符串与数字的转换：
‘123abc’ 被转换为数字 123，而 ‘abc123’ 会被转换为 0

利用PHP的类型转换特性，可以在绕过输入验证或防护机制时，利用 == 比较或者在isset() 或 empty() 等函数的判断中绕过一些条件检查

  1. SQL注入中的类型转换
    MySQL中的类型转换：
    在SQL注入时，MySQL会对不同数据类型进行隐式转换，导致一些有趣的行为
    例如：1 == ‘1’ 在SQL中为真，因此可以用 ‘1’ 替代 1 来绕过某些类型检查
    还可以通过 UNION SELECT 技巧，利用隐式转换将非数值类型转换为数值，进行SQL注入攻击
    例子：
    SELECT * FROM users WHERE id = ‘1’ OR 1=1 可以通过类型转换绕过某些验证，’1’ 会被转为数值 1
    有时，数字字段输入 ‘1’ 可以绕过对数字字段的类型检查，从而导致SQL注入

  2. 布尔型注入（Boolean-based Blind SQL Injection）
    在某些情况下，数据库会对注入的条件进行类型转换，这可能导致布尔型条件的行为被绕过或伪造
    比如，1 OR 1=1 和 1 OR ‘a’=’a’ 都可以通过类型转换绕过SQL查询中的条件判断

  3. XPath注入中的弱类型问题
    XPath是一种在XML文档中查找信息的语言，在某些情况下，XPath语句中的类型转换也可能被攻击者利用来注入恶意内容
    例如，如果传入的变量是字符串类型，可能会被强制转换成数字，导致查询条件被绕过

  4. Web应用防火墙（WAF）绕过
    一些WAF或输入验证机制依赖于特定类型的比较和匹配方式，弱类型特性可能帮助绕过这些机制。例如，WAF检测到’ OR ‘1’=’1时，可能会认为这是非法输入，但通过使用弱类型特性，如’ OR 1=1，可能可以绕过WAF

  5. 绕过JSON解析
    在某些情况下，输入数据可能以JSON格式进行传输。如果没有严格类型检查，攻击者可以利用JSON的自动类型转换特性来构造恶意请求
    例如，”0”（字符串）可能被转换为 false，而 “1” 被视为 true，这种特性可以被利用来绕过验证或逻辑判断

SQL注入

  1. SQL注入概述
    SQL注入攻击是指攻击者通过将恶意的SQL代码嵌入到应用程序的输入字段（如登录框、搜索框等）中，迫使应用程序执行该恶意SQL语句，从而影响数据库的正常操作，甚至获取数据库的敏感信息。

  2. SQL注入的分类
    基于错误的SQL注入（Error-based SQLi）：
    攻击者通过故意触发SQL错误，获取错误信息来推测数据库的结构和表字段。这通常发生在应用程序没有处理数据库错误的情况下。

联合查询（Union-based SQLi）：
通过在SQL查询中使用UNION操作符，攻击者可以将恶意的查询结果与正常查询的结果合并，从而获取数据库中的其他数据。
盲注（Blind SQLi）：

攻击者无法直接从错误信息中获得反馈，需通过观察网页响应的变化来推测数据库的内容。盲注有两种常见方式：
布尔盲注（Boolean-based Blind SQLi）：通过向查询中插入条件（如1=1或1=2）来观察页面的响应变化。

时间盲注（Time-based Blind SQLi）：通过插入导致延迟的查询（如SLEEP函数）来推测数据库的存在信息。

时间延迟注入（Time-based SQLi）：
攻击者通过在查询中插入时间延迟函数（如SLEEP(5)）来判断条件的真假，进一步探索数据库信息。
3. SQL注入攻击的基本步骤
识别漏洞：首先，攻击者需要识别目标网页是否存在SQL注入漏洞，通常通过对输入字段进行一些特殊字符测试（如’、”等）来观察系统的反应。

分析数据库结构：一旦确认存在SQL注入漏洞，攻击者会尝试获取有关数据库的信息，例如表名、字段名等。这可以通过各种方式实现，如利用错误消息、联合查询等。

执行攻击：获取足够的信息后，攻击者可以进行数据获取、删除、修改等操作，甚至可以通过SQL注入获取管理员权限。

  1. SQL注入的常见利用
    获取数据库信息：

sql
‘ OR 1=1 –
上述查询会导致SQL语句的条件永远为真，攻击者可以利用该漏洞绕过认证过程。

联合查询注入：
利用UNION查询合并恶意查询的结果。例如，获取第二个表的所有数据：

sql
‘ UNION SELECT null, username, password FROM users –
盲注：

布尔盲注：
sql
‘ AND 1=1 – (返回正常页面)
‘ AND 1=2 – (返回错误页面)
时间盲注：
sql
‘ AND SLEEP(5) – (查询延迟5秒)

  1. 防御SQL注入
    使用预处理语句（Prepared Statements）：
    预处理语句使用绑定参数的方式，将用户输入与SQL查询分开处理，避免了SQL注入风险。常见的编程语言支持预处理语句，如PHP的PDO、Python的MySQL Connector等。
    python
    cursor.execute(“SELECT * FROM users WHERE username = %s AND password = %s”, (username, password)).

输入验证与过滤：
对所有用户输入进行严格的验证，过滤掉危险字符，如’、”、–、;等。可以使用正则表达式限制输入的格式。

使用最小权限原则：
数据库账户应当仅授予其执行必要操作的权限。避免使用具有管理员权限的数据库账户来运行应用程序。

错误处理：
避免在应用程序中显示详细的数据库错误信息。如果出现错误，应该显示通用的错误信息，而不暴露数据库结构。

SQL查询审计与监控：
定期审计和监控数据库的SQL查询日志，发现并阻止异常的查询请求。

  1. CTF Web中的SQL注入题目常见技巧
    错误消息利用：有时，目标网站可能没有处理SQL错误，攻击者可以通过错误信息获得有关数据库的信息。例如，利用’导致SQL错误，获取数据库的错误提示。

巧妙构造Payload：CTF题目常常要求用特殊的SQL语句来读取特定的数据或绕过认证。例如，构造特定的SQL查询来查询隐藏的管理员账户信息。

多种注入方式结合：有时，攻击者需要结合多种SQL注入技术，如先利用盲注推测数据库结构，再使用联合查询获取数据。

web文件上传

  1. 文件上传基本原理
    文件上传漏洞：允许用户将文件上传到服务器，若未对上传的文件进行足够的验证或限制，攻击者可以上传恶意文件执行命令或获取敏感信息。
    文件上传机制一般包括前端表单和后端接收上传的处理逻辑。
  2. 常见的文件上传漏洞类型
    文件类型验证绕过：后端往往通过检查文件扩展名（例如 .jpg, .png）来验证文件类型，但攻击者可以通过修改文件扩展名绕过这种验证。
    例如，可以将 .php 文件改为 .jpg，上传时修改扩展名，后端可能会错过实际文件类型的检测。最常见的就是一句话木马。
    MIME 类型检查绕过：有时后端通过检查 MIME 类型来判断文件类型。攻击者可能通过篡改文件头部的 MIME 类型来绕过后端的检查。
    例如，修改文件的 Content-Type 为 image/jpeg，即便文件本身是一个恶意 PHP 脚本。
    路径遍历攻击：上传的文件会被保存在服务器上，如果没有对文件路径进行严格限制，攻击者可以利用路径遍历漏洞上传文件到敏感位置，甚至执行服务器上的文件。
    例如，上传文件名为 ../../../../etc/passwd，文件可能被上传到 /etc/passwd 文件路径，导致信息泄露。
    执行权限绕过：即使文件成功上传到服务器，若文件存放目录没有执行权限，攻击者可能也无法执行上传的恶意文件。但某些服务器配置不当时，可能会允许文件执行。
    例如，将文件上传到可执行目录（如 cgi-bin）中，攻击者可以通过浏览器访问并执行上传的 PHP 或其他脚本文件。
    恶意文件上传：攻击者上传的文件可能是恶意的 Web 脚本（如 PHP、ASP、JSP 等），可以在服务器上执行任意代码。
    例如，上传一个 PHP 文件，通过 HTTP 请求访问该文件，执行任意系统命令。3. 防御文件上传漏洞的常见策略
    文件类型验证：在后端对文件进行严格的类型验证。除了扩展名，使用如 getimagesize() 等函数验证文件的 MIME 类型和文件内容是否与期望一致。
    限制文件大小：限制文件上传的最大大小，防止攻击者通过上传大量数据导致服务拒绝（DoS）攻击。
    文件名规范化：对上传的文件名进行规范化处理，避免路径遍历或特殊字符导致安全漏洞。
    文件存储位置：将上传的文件存储在不可执行的目录中，避免上传的恶意脚本被执行。通常可以设置一个专门的文件存储目录，并设置为不可执行。
    使用随机文件名：为上传的文件生成随机文件名，避免攻击者通过文件名直接访问或替换文件。
    扫描恶意内容：使用杀毒软件或扫描工具检查上传的文件是否包含恶意代码或病毒。
  3. 常见的绕过技巧
    文件扩展名绕过：
    攻击者可以上传以 .jpg、.png、.gif 等常见扩展名的文件，实际内容是恶意代码。服务器只检查扩展名而忽略了文件内容。
    另一种常见绕过方式是通过文件双扩展名（如 evil.php.jpg）绕过检查。
    利用图像文件上传脚本：
    上传一个看似图像文件的 PHP 脚本，例如通过上传带有伪装的图像文件（如 image.php.jpg），并利用文件内容（即 PHP 脚本）进行攻击。
    利用混淆手段绕过 MIME 类型检查：
    通过修改文件头（magic bytes）或添加额外的伪造数据（例如在图片文件头部插入 PHP 代码），从而绕过文件内容检查。
    Path Traversal 和 RCE：
    攻击者可能尝试通过上传文件到不安全的路径（如 /tmp 或其他目录），并通过访问该文件实现远程代码执行。
  4. 漏洞利用过程示例
    假设某 Web 应用允许用户上传图片，并且后端只对扩展名做了简单的验证：

第一步：分析上传功能：确认文件类型验证和路径限制是否足够严格。
第二步：绕过文件类型验证：尝试上传 PHP 文件并通过更改扩展名（如 malicious.php.jpg）绕过检查。
第三步：路径遍历攻击：尝试上传文件到不可访问的目录，例如通过文件名 ../../../../var/www/html/shell.php 来访问系统关键路径。
第四步：利用 RCE 执行代码：通过上传 PHP 文件到可执行目录并直接访问，执行攻击代码。
